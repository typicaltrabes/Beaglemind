    2. **shared.ts** -- Add `pushSubscriptions` table to shared schema:
       ```typescript
       export const pushSubscriptions = shared.table('push_subscriptions', {
         id: uuid('id').primaryKey().defaultRandom(),
         userId: text('user_id').notNull().references(() => users.id),
         endpoint: text('endpoint').notNull(),
         keysP256dh: text('keys_p256dh').notNull(),
         keysAuth: text('keys_auth').notNull(),
         createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
       });
       ```
       Import `users` from auth-schema if needed for the reference. Add unique constraint on (userId, endpoint) to avoid duplicate subscriptions.

    3. **apps/web/app/api/push/subscribe/route.ts** -- POST endpoint:
       - Receives `{ subscription: PushSubscription }` from client
       - Gets current user from auth session
       - Upserts into push_subscriptions table (on conflict userId+endpoint, do nothing)
       - Returns 200

    4. **apps/web/lib/push-client.ts** -- Client-side push utilities:
       - `subscribeToPush()`: Requests notification permission, gets push subscription from service worker registration using VAPID public key (from `NEXT_PUBLIC_VAPID_PUBLIC_KEY` env var), POSTs to `/api/push/subscribe`
       - `isPushSupported()`: Checks for `'serviceWorker' in navigator && 'PushManager' in window`

    5. **apps/web/components/push-permission.tsx** -- Client component:
       - Renders a small "Enable notifications" button/banner (only if push is supported and not already subscribed)
       - On click, calls `subscribeToPush()`
       - After successful subscription, hides itself
       - Check existing subscription via `registration.pushManager.getSubscription()`
       - Mount this component inside DashboardShell (after the header, visible on mobile only via `md:hidden`)

    6. **apps/agent-hub/src/notifications/push-service.ts**:
       - Import `web-push` and configure with VAPID keys from env (VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_SUBJECT)
       - Export `sendPushNotification(tenantId: string, payload: { title: string; body: string; url: string })`:
         - Query push_subscriptions joined with organization members to find all users in the tenant
         - For each subscription, call `webpush.sendNotification(subscription, JSON.stringify(payload))`
         - On 410 (Gone) response, delete the stale subscription
       - Export `notifyPlanApproval(tenantId: string, runId: string, projectId: string, planName: string)`:
         - Calls sendPushNotification with title "Mo needs approval", body "[planName]", url `/projects/${projectId}/runs/${runId}`
       - Export `notifyQuestion(tenantId: string, runId: string, projectId: string, agentName: string, questionPreview: string)`:
         - Calls sendPushNotification with title `Question from ${agentName}`, body truncated questionPreview, url `/projects/${projectId}/runs/${runId}`

    7. **message-router.ts** -- Hook into event processing:
       - After persisting a `plan_proposal` event, call `notifyPlanApproval(tenantId, runId, projectId, planName)`
       - After persisting a `question` event, call `notifyQuestion(tenantId, runId, projectId, agentName, questionText)`
       - Import push-service functions. Guard with try/catch so push failures never break the event pipeline.
       - Extract projectId from the run record (query runs table by runId to get projectId) -- cache this lookup if needed.

    Use env vars: VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_SUBJECT for the Hub. NEXT_PUBLIC_VAPID_PUBLIC_KEY for the web client.

| T-10-04 | I (Info Disclosure) | Push notification body | mitigate | Notification body contains plan name / question preview only -- no sensitive content. Truncate to 100 chars. |
| T-10-05 | D (Denial of Service) | Push send on every event | accept | Low volume (governance events only, not every message). Rate limiting not needed for Year 1. |
