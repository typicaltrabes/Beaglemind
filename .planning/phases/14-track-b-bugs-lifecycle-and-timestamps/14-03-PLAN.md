    /**
     * Phase 13 schema migration (idempotent), extended in Phase 14.
     *
     * Adds two additive columns:
     *   1. tenant_<id>.runs.title           VARCHAR(80) NULL
     *   2. shared.users.preferences         JSONB NOT NULL DEFAULT '{}'::jsonb
     *
     * Phase 14 additions (UAT-14-03):
     *   - Tenant discovery now reads from shared.organizations (the real source-of-truth

     *     which inserts into shared.organizations with id = tenant.id).
     *   - One-shot backfill: marks any `pending` run older than 1 day as `cancelled`,
     *     leaving fresh `pending` rows alone (none expected today; both POST /api/runs
     *     and POST /api/runs/[id]/messages default to `executing`, so this is pure
     *     data debt cleanup).
     *
     * All operations use ADD COLUMN IF NOT EXISTS / status='pending' AND age>1day predicates,
     * so re-running is a no-op on the second invocation.
     * Per CONTEXT.md: "Both additive, nullable / defaulted, so no downtime risk."
     */
    export async function migratePhase13() {
      const client = postgres(connectionString, { max: 1 });
      const db = drizzle(client);

      try {
        // 1. Add preferences column to shared.users (single statement).
        await db.execute(sql`
          ALTER TABLE shared.users
          ADD COLUMN IF NOT EXISTS preferences jsonb NOT NULL DEFAULT '{}'::jsonb
        `);
        console.log('Added shared.users.preferences');

        // 2. Iterate every tenant schema. Source-of-truth is shared.organizations
        //    (NOT shared.tenants — that table is empty by convention; see
        //    provision-tenant.ts which inserts org.id = tenant.id).
        const tenantRows = (await db.execute(
          sql`SELECT id FROM shared.organizations`,
        )) as unknown as { id: string }[];

        for (const row of tenantRows) {
          // Sanitize tenant id the same way createTenantSchema does (hyphen -> underscore).
          const schemaName = `tenant_${row.id.replace(/-/g, '_')}`;
          try {
            // 2a. Phase 13: add the runs.title column.
            await db.execute(sql`
              ALTER TABLE ${sql.identifier(schemaName)}.runs
              ADD COLUMN IF NOT EXISTS title varchar(80)
            `);
            console.log(`Added ${schemaName}.runs.title`);

            // 2b. Phase 14 (UAT-14-03): cancel orphan `pending` runs older than 1 day.
            // Idempotent: rerunning matches 0 rows the second time.
            const cancelResult = await db.execute(sql`
              UPDATE ${sql.identifier(schemaName)}.runs
              SET status = 'cancelled', updated_at = NOW()
              WHERE status = 'pending' AND created_at < NOW() - interval '1 day'
            `);
            const cancelled = (cancelResult as unknown as { count?: number }).count ?? 0;
            console.log(`Backfilled ${cancelled} orphan pending → cancelled in ${schemaName}.runs`);
          } catch (err) {
            // One bad tenant must not block the rest. Log and continue per CONTEXT.md
            // failure model ("write to log and continue — the next run will still work").
            console.error(`Failed migration step for ${schemaName}:`, err);
          }
        }

        console.log(`Phase 13 + 14 migration complete (${tenantRows.length} tenants).`);
      } finally {
        await client.end();
      }
    }

    // Allow direct invocation: pnpm --filter @beagle-console/db exec tsx src/scripts/migrate-13.ts
    if (import.meta.url.endsWith(process.argv[1] ?? '')) {
      migratePhase13().catch((err) => {
        console.error('Phase 13 migration failed:', err);
        process.exit(1);
      });
    }

    Implementation notes — required reading before writing the diff:

    - **shared.organizations vs shared.tenants.** STATE.md "Pending Todos" item #2 logs the bug exactly: "Script iterates `shared.tenants` (empty by convention in this project) instead of `shared.organizations` (the real source-of-truth). The provisioning code at `packages/db/src/provision-tenant.ts:117-122` is the canonical proof — it inserts org.id = tenant.id, so iterating organizations gives the same tenant IDs that runtime code uses to look up schemas.
    - **Hyphen→underscore is unchanged.** `organization.id` is a `text` column but its values come from `shared.tenants.id` which is a UUID — the existing `row.id.replace(/-/g, '_')` works without modification. Verify by reading provision-tenant.ts:117 (`id: tenant.id`).
    - **Per-tenant try/catch added.** The original script crashes the whole script on any single tenant's DDL failure (no try/catch in the loop body). With the new backfill UPDATE, a tenant whose runs table somehow doesn't exist or lacks the column would block all subsequent tenants. Wrap each tenant's body in try/catch and log+continue. The outer try/finally remains for client cleanup.
    - **Why `updated_at = NOW()` and not `NOW() AT TIME ZONE 'UTC'`.** The column is `timestamp with time zone` (per tenant.ts:29), which Postgres stores as UTC regardless of the input value's tz; `NOW()` returns `timestamp with time zone` — types match. The Drizzle camelCase `updatedAt` does NOT apply to raw SQL — the actual column name in Postgres is `updated_at`.
    - **Why `interval '1 day'` and not e.g. `'1 hour'`.** Per CONTEXT.md `<decisions>` Bug 3: "leaving new short-lived `pending` rows alone." A run created today by some unforeseen code path (e.g. a test script) shouldn't be reaped before it can transition; 1 day is enough to catch the 18 historical orphans and any pre-Phase-4 data without touching today's traffic.
    - **The `cancelResult.count` extraction.** `db.execute(sql\`UPDATE ...\`)` with the `postgres-js` driver returns a result object with a `count` property (number of rows updated). Coerce safely with `?? 0` because TypeScript can't narrow the unknown shape. The console line is informational; if it logs `0` for every tenant on first run that's a flag worth investigating.
    - **The Phase 13 console.log message** stays `Added ${schemaName}.runs.title` — do not change to "Phase 13: Added..." or similar; CI / deploy logs from Phase 13 may grep for the original string.
    - **The terminal completion log** changes from `Phase 13 migration complete (${tenantRows.length} tenants).` to `Phase 13 + 14 migration complete (${tenantRows.length} tenants).` — explicit naming so an operator running the script knows both phases' migrations applied.
