<objective>
Close UAT-12-04 — the run page header currently shows only the run UUID, so when switching tabs or returning to a run the user cannot tell what the run is about without scrolling. Add:
1. A `GET /api/runs/[id]` endpoint that returns `{ id, prompt, status, kind, createdAt, updatedAt }` for a single run, scoped to the caller's tenant.
2. A `useRun(runId)` TanStack Query hook that fetches and caches the result.
3. An `<h1>` title block on the run page rendering the prompt as a one-line truncated title above the existing status badge + UUID row, with `title` attribute = full prompt and a fallback "Untitled run" when prompt is null.

Purpose: Per CONTEXT.md `<decisions>` §Defect 4 — "the run header shows the user's prompt as a one-line truncated title above … the existing status chip + UUID row, so the run is identifiable at a glance." Run-history endpoint returns a list, not a single run. So the smaller diff is to add the new route — three small files vs. extending the SSE stream endpoint which is a separate concern.

    ```typescript
    import { NextResponse } from 'next/server';
    import { requireTenantContext, getTenantDb } from '@/lib/get-tenant';
    import { eq } from 'drizzle-orm';

    export const runtime = 'nodejs';

    export async function GET(
      _request: Request,
      { params }: { params: Promise<{ id: string }> },
    ) {
      try {
        const { id } = await params;
        const { tenantId } = await requireTenantContext();
        const { db: tdb, schema } = getTenantDb(tenantId);

        const rows = await tdb
          .select({
            id: schema.runs.id,
            projectId: schema.runs.projectId,
            prompt: schema.runs.prompt,
            status: schema.runs.status,
            kind: schema.runs.kind,
            createdAt: schema.runs.createdAt,
            updatedAt: schema.runs.updatedAt,
          })
          .from(schema.runs)
          .where(eq(schema.runs.id, id))
          .limit(1);

        const run = rows[0];
        if (!run) {
          return NextResponse.json({ error: 'Run not found' }, { status: 404 });
        }

        return NextResponse.json(run);
      } catch (error) {
        console.error('GET /api/runs/[id] error:', error);
        return NextResponse.json(
          { error: 'Internal server error' },
          { status: 500 },
        );
      }
    }
    ```

    Why these specific fields: matches the columns the run-history list returns + adds nothing the run page doesn't need. Keeps the response payload small (~200 bytes typical) so TanStack Query caching is cheap.

    Tenant isolation: `requireTenantContext` returns the caller's tenantId; `getTenantDb(tenantId)` returns a Drizzle client scoped to the tenant's schema. The `.where(eq(schema.runs.id, id))` is scoped to that schema, so a cross-tenant read with a known UUID returns 404 (not the row from another tenant). Same pattern the history endpoint uses.

    B) Create `apps/web/lib/hooks/use-run.ts` with EXACTLY this content:

    ```typescript
    'use client';

    import { useQuery } from '@tanstack/react-query';

    export interface Run {
      id: string;
      projectId: string;
      prompt: string | null;
      status: string;
      kind: string;
      createdAt: string;
      updatedAt: string;
    }

    export function useRun(runId: string) {
      return useQuery<Run>({
        queryKey: ['runs', 'detail', runId],
        queryFn: async () => {
          const res = await fetch(`/api/runs/${runId}`);
          if (!res.ok) throw new Error('Failed to fetch run');
          return res.json();
        },
        enabled: !!runId,
        staleTime: 60_000,
      });
    }
    ```

    Notes on the hook:
    - `staleTime: 60_000` (1 minute) — the prompt never changes after a run is created, but other fields (`status`, `updatedAt`) do change as the run progresses. SSE already updates `useRunStore.status`, so the run page reads `status` from the SSE-driven store, NOT from this hook. The hook is effectively a one-shot prompt fetcher; 60s staleTime is just hygiene to avoid re-fetching on tab switches.
    - `enabled: !!runId` — guards against the React-render-during-loading edge case.
    - `queryKey: ['runs', 'detail', runId]` — namespaced under `['runs', ...]` so it shares an invalidation prefix with `['runs', 'history', ...]` should we ever need to broadcast-invalidate.

    DO NOT:
    - Add a Zod schema validating the response — run-history doesn't, this hook follows the same pattern. If type drift becomes a problem, add validation in a follow-up.
    - Add a mutation hook (this is read-only).
    - Pre-populate the query cache from `useRunStore` — the store is event-stream state, not run-record state. Keep them separate.
    - Add SSE invalidation. The hook is fire-once per run page visit; that's enough for the title use case.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP → API route | `runId` from URL path is read into the `params` promise and used in a `where(eq(schema.runs.id, id))` query. |
| DB row → UI | `run.prompt` flows from DB to React text node `{titleDisplay}` and to the `title` attribute. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-04 | Spoofing / cross-tenant access | `GET /api/runs/[id]` | mitigate | Route calls `requireTenantContext()` first (throws/redirects on unauth) and uses `getTenantDb(tenantId)` to scope the query to the caller's schema. A request for a runId that exists in another tenant returns 404 because the query runs against the caller's `schema.runs` table — same pattern as the existing `/api/runs/history` route. |
| T-12-05 | Tampering / SQL injection via runId path param | drizzle `eq(schema.runs.id, id)` | mitigate | drizzle-orm's `eq` parameterizes the value — the runId is bound, not interpolated. Same pattern used elsewhere in the codebase for tenant-scoped lookups. |
| T-12-06 | Info disclosure (XSS via prompt content) | run page `<h1>` and `title` attribute | mitigate | `{titleDisplay}` is rendered as a React text node — React HTML-escapes by default. `title={titleDisplay}` renders the prompt into an HTML attribute; React escapes attribute values, so a prompt containing `"` or `<` cannot break out of the attribute. No `dangerouslySetInnerHTML`. |
| T-12-07 | Repudiation | (n/a) | accept | Read-only endpoint, no state change. |
| T-12-08 | Denial of service | `GET /api/runs/[id]` | accept | Simple parameterized PK lookup with `LIMIT 1`. Negligible cost. Same surface as the existing read endpoints. |
</threat_model>
