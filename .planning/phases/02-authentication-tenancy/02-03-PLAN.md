<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| invite link -> accept-invite API | Untrusted invitation ID from URL |
| TOTP input -> auth API | User-provided 6-digit code |
| provisioning script -> database | Admin creates tenants with full DB access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Spoofing | accept-invite route | mitigate | Invitation ID is cryptographic (Better Auth generated). Validate invitation exists and is not expired before creating user. |
| T-02-09 | Elevation of Privilege | accept-invite route | mitigate | Server-side signUpEmail only runs after invitation validation. No way to create account without valid invitation per D-05. |
| T-02-10 | Information Disclosure | mfa-setup page | mitigate | TOTP secret and backup codes shown once, stored encrypted by Better Auth. Page warns user to save backup codes. |
| T-02-11 | Spoofing | mfa-challenge page | mitigate | Better Auth enforces TOTP verification with time-window tolerance. Backup codes are single-use. |
| T-02-12 | Information Disclosure | provision-tenant.ts | accept | Provisioning script runs server-side only (CLI). Admin credentials passed as arguments, not stored in code. Password is bcrypt-hashed before DB insert. |
| T-02-13 | Information Disclosure | tenant isolation | mitigate | Integration test (D-10) verifies cross-tenant data is invisible. Schema-level isolation via PostgreSQL search_path. |
</threat_model>
