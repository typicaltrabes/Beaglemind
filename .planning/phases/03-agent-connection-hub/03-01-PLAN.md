**3. Create `apps/agent-hub/src/connections/reconnect.ts`** per D-04:
- Export `calculateBackoff(attempt: number, base = 1000, max = 30000, jitter = 0.5): number`
- Formula: exponential = min(base * 2^attempt, max), jitterRange = exponential * jitter, result = exponential + (random * 2 - 1) * jitterRange, clamp to >= 0
- Export `ConnectionState` type = 'connecting' | 'connected' | 'disconnecting' | 'reconnecting' | 'closed'

**4. Create `apps/agent-hub/src/connections/managed-connection.ts`** per D-01, D-03, D-04, D-13:
- Class `ManagedConnection` wrapping a single ws WebSocket connection
- Constructor takes: agentId, url, pingIntervalMs, pongTimeoutMs, onMessage callback, onStateChange callback
- `connect()`: creates new WebSocket, sets up open/message/pong/close/error handlers
- On open: state -> connected, start ping/pong interval, call onStateChange
- On message: parse JSON, call onMessage(agentId, parsed)
- On pong: clear pong timeout
- On close: stop ping/pong, state -> reconnecting, schedule reconnect via setTimeout(connect, calculateBackoff(attempt))
- On error: log only (ws emits close after error -- do NOT reconnect on error per RESEARCH.md Pitfall 1)
- `send(message: object)`: JSON.stringify and send if readyState === OPEN
- `close()`: state -> disconnecting, close WebSocket gracefully
- `getState()`: returns current ConnectionState
- Private ping/pong: setInterval ping, setTimeout for pong timeout -> terminate on no pong
- Track reconnect attempt counter, reset to 0 on successful open
- Cap outbound queue at 100 messages during reconnection per RESEARCH.md Pitfall 3

**5. Create `apps/agent-hub/src/connections/agent-registry.ts`** per D-03:
- Class `AgentRegistry` containing Map<string, ManagedConnection>
- Constructor takes agent config array from config.ts, plus onMessage and onStateChange callbacks
- `connectAll()`: creates ManagedConnection for each agent and calls connect()
- `send(agentId, message)`: routes message to specific agent's connection
- `getStatus()`: returns object with each agent's id, url, and connection state
- `getHealthStatus()`: returns { status: 'ok'|'degraded', agents: { [id]: state } } — 'ok' if all connected, 'degraded' if any not connected
- `closeAll()`: gracefully close all connections

The POST endpoints are placeholders because event persistence and Redis publishing (Plan 03) must exist before they can function. The Hub starts, connects to agents, and reports health. This is the connection layer only.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-01 | Tampering | Inbound OpenClaw messages | mitigate | Validate all inbound messages with OpenClawInbound Zod schema in managed-connection.ts. Drop and log invalid frames. |

| T-03-04 | Denial of Service | Reconnection storms | mitigate | Exponential backoff + jitter (D-04) prevents reconnection storms. Max 30s between attempts. |
