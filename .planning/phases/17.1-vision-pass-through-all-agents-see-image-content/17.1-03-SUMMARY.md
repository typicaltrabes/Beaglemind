## Decisions Made

1. **Outcome D — OpenClaw agent CLI has no vision flag.** Investigation against `/usr/lib/node_modules/openclaw/dist/register.agent-DA0Frq4g.js` is dispositive: the entire option chain is text-only (`-m, --message`, `-t, --to`, `--session-id`, `--agent`, etc., 14 options total, zero image-related). Inline data-URI / base64 in `--message` also unsupported. Per the plan's outcome-D branch, we ship the entire upstream/downstream pipeline anyway with a single log-and-skip at the bridge seam, so the day OpenClaw ships a vision flag, only the bridge body changes.

2. **VISION_CAPABLE hardcoded in routes.ts (PATTERNS Option B), not imported from web or shared.** Cross-package import for one boolean is excessive; the symbol mirrors `apps/web/lib/agent-config.ts` and a comment marks the duplication. Promotion to `@beagle-console/shared` is deferred until a third consumer appears (PATTERNS.md "Note" at lines 387-393).

3. **HubImageAttachment Zod inline in routes.ts (PATTERNS Option A), not in `@beagle-console/shared`.** Single consumer (the hub validates, the bridge consumes); no other package needs the symbol. Same defer-to-shared rationale as VISION_CAPABLE.

4. **Image-bytes budget enforced web-side AS WELL AS hub Zod max(4).** The web-side 10 MB total-bytes budget is the cheap shed-load (skip the `Promise.all` of `GetObjectCommand`s); the hub `.max(4)` is the malformed-payload defense (rejects an attacker sending 50 image attachments in one request before any bridge work runs). Belt-and-suspenders is correct here — the failure modes are different.

5. **`imageAttachments` declared at route-handler scope, lifted above the `if (attachmentIds.length > 0)` block.** The variable must be visible to the `hubClient.startRun` call after the conditional. Lifting once is cleaner than declaring twice or mutating. The `let imageAttachments: Array<...> | undefined` annotation gives the explicit types so callers know it's "either an array or undefined."

6. **Image rows filtered from the existing `ordered` array, not re-queried.** Plan 17.1-02 already produces `ordered` (Map-reordered, user-selection-order-preserving array of artifact rows). Filtering for `mimeType.startsWith('image/')` is the cheap derivation; re-querying would be redundant and would break the user-selection-order invariant.

7. **Parse logic in openclaw-cli-bridge.ts preserved verbatim from line 35 onwards.** The diff confirms exactly: 2 changes (signature widening + warn block) plus 1 cosmetic (trailing-whitespace cleanup on a previously-blank-with-spaces line). No behavior change to the existing JSON-parse + cost-estimation path. This is the explicit Task 3 acceptance criterion and a safety property — the bridge's parse logic has been load-bearing since Phase 9.
