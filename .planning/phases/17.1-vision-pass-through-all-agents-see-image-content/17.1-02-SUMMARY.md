## Decisions Made

1. **Image branch hoisted ahead of extractedText branch (not co-conditional).** The plan called for `if (r.mimeType.startsWith('image/'))` to come before `else if (r.extractedText)`, which I implemented faithfully. This means even if a defective future code path were to write `extractedText` to an image row, `description` still wins for image mimes — by spec. The defensive test case `'image description takes precedence over extractedText'` locks this in.

2. **`description: string | null` is required, not optional.** Every Drizzle SELECT that feeds `buildAttachmentBlock` must explicitly project the column. Optional `description?: string | null` would let consumers silently drop the field and get `undefined` (which the implementation would treat as falsy → fallback string), but it would also hide schema-projection bugs. Required-but-nullable forces consumers to think.

3. **Test fixtures carry explicit `description: null` on every object.** TypeScript strict property checking would catch a missing field anyway; this convention surfaces the description axis to test authors. When someone adds a new test case in a future plan, they'll see every existing fixture has a `description` field and understand it's part of the contract.

4. **`minioKey` left out of the messages-route projection.** Plan 17.1-03 (image base64 fetch) owns it. Keeping this plan's diff to a single-field addition prevents merge conflict risk with 17.1-06 (transcript UX), which also touches `messages/route.ts` later in Wave 2.

## Threat Model Status

All three threats in the plan's `<threat_model>` register carry forward as-mitigated:

- **T-17-1-02-01 (Prompt-injection via description → agent prompt):** mitigated. The `--- USER ATTACHMENTS ---` / `--- END ATTACHMENTS ---` sentinels delineate user-controlled content. The `Description:` prefix telegraphs that the body line is machine-generated from user-uploaded image. The system-level groupContext in `apps/agent-hub/src/http/routes.ts` already instructs agents that user content is untrusted. Same disposition as T-17-03-04 (PDF extracted text).
- **T-17-1-02-02 (Description leak across runs):** mitigated. The SELECT WHERE clause is unchanged: `inArray(artifacts.id, attachmentIds) AND eq(artifacts.runId, runId) AND eq(artifacts.agentId, 'user')`. Cross-run reuse fails the strict-ownership 404 check (`rows.length !== attachmentIds.length`). Description is per-artifact-row; Plan 17.1-02 added no new query path.
- **T-17-1-02-03 (Description quality / agents misled):** accepted. The artifacts row is the audit trail (`createdAt` + `description` text). UAT-17-1-01 in Plan 17.1-04 verifies the description reaches agents. Description-quality issues (e.g., Haiku misreading the image) are model-quality concerns; planner's note "iterate prompt during UAT" applies.

No new threat surface introduced — this plan widened a SELECT projection (no new query) and added a body-line emission path (text only, no new I/O surface).
