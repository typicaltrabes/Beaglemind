<objective>
Fix DEFECT-17-C: agents go silent or context-blind when a follow-up message is sent to a `completed` run. Two changes:
1. Hub's `runRoundTable` loads prior conversation events from the EventStore and injects them as a `--- PRIOR CONVERSATION ---` block — agents now see the full context for the run.
2. When an agent fails (CLI bridge error, timeout), surface an `agent_error` event so the user sees a one-line marker in the transcript instead of silence.

Web side already does the right thing — it flips status from `completed` to `executing` and re-calls `hubClient.startRun`. Composer already permits sending on completed runs. No web changes needed beyond what 17.1-06 already does.
</objective>

Add a `list` method to `EventStore`:

```ts
async list(
  tenantId: string,
  runId: string,
  opts?: { limit?: number },
): Promise<HubEventEnvelope[]> {
  const { events } = createTenantSchema(tenantId);
  const limit = opts?.limit ?? 30;

  // Drizzle: select where runId = runId, order by sequenceNumber DESC, limit,
  // then reverse to return ascending order.
  const rows = await this.db
    .select()
    .from(events)
    .where(eq(events.runId, runId))
    .orderBy(desc(events.sequenceNumber))
    .limit(limit);

  return rows.reverse().map((r: any) => ({
    type: r.type,
    agentId: r.agentId,
    runId: r.runId,
    tenantId,
    content: r.content,
    metadata: r.metadata ?? {},
    sequenceNumber: r.sequenceNumber,
    timestamp: r.createdAt instanceof Date ? r.createdAt.toISOString() : r.createdAt,
  }));
}
```

Modify `runRoundTable` in `apps/agent-hub/src/http/routes.ts`:

1. Accept an `eventStore: EventStore` parameter (thread it through from `handleRunStart` / `index.ts`).
2. **Before** the `for (const agentId of agents)` loop, fetch prior history:
   ```ts
   const HISTORY_EVENT_LIMIT = 30;
   const HISTORY_CHAR_BUDGET = 80_000;
   const priorEvents = await eventStore.list(tenantId, runId, { limit: HISTORY_EVENT_LIMIT });
   // Drop the most-recent event if it's the user prompt we just persisted —
   // we don't want it in PRIOR CONVERSATION; it goes in "User: ..." below.
   const filtered = priorEvents.filter(
     (e) => e.type === 'agent_message',
   );
   // The current user message was persisted with sequenceNumber === userEvent.sequenceNumber;
   // identify and exclude.
   const history = filtered.filter((e) => e.sequenceNumber !== currentUserSequence);

   // Build the formatted history string with a char budget — drop oldest first.
   let historyBlock = '';
   if (history.length > 0) {
     const lines = history.map((e) => {
       const speaker =
         e.agentId === 'user'
           ? 'User'
           : e.agentId.charAt(0).toUpperCase() + e.agentId.slice(1);
       const text = (e.content as { text?: string }).text ?? '';
       return `${speaker}: ${text}`;
     });
     // Trim oldest until under budget.
     while (lines.join('\n\n').length > HISTORY_CHAR_BUDGET && lines.length > 0) {
       lines.shift();
     }
     historyBlock = `--- PRIOR CONVERSATION ---\n\n${lines.join('\n\n')}\n\n--- END PRIOR CONVERSATION ---\n\n`;
   }
   ```
   `currentUserSequence` is the sequenceNumber returned from the `userEvent` persistence in `handleRunStart` — pass it down to `runRoundTable`.

3. Inject `historyBlock` into BOTH branches of the per-agent prompt construction:
   ```ts
   if (transcript.length === 0) {
     fullPrompt = `${groupContext}\n\n${historyBlock}User: ${userPrompt}\n\n${displayName}, you're first to respond.`;
   } else {
     fullPrompt = `${groupContext}\n\n${historyBlock}--- GROUP DISCUSSION ---\n\nUser: ${userPrompt}\n\n${transcript.join('\n\n')}\n\n--- YOUR TURN ---\n\n${displayName}, respond to the discussion above. Reference other agents by name when you agree or disagree.`;
   }
   ```

4. **Failure surfacing.** In the existing `} catch (err: any) { ... }` block where `sendToAgent` failures are logged, ALSO publish an `agent_error` event so the transcript shows something:
   ```ts
   } catch (err: any) {
     log.error({ error: err.message, runId, agentId }, 'Agent failed in round-table, continuing');
     try {
       await router.persistAndPublish(tenantId, {
         type: 'agent_message',
         agentId,
         runId,
         tenantId,
         content: {
           text: `(${agentId} failed to respond — see logs for details)`,
         },
         metadata: { errorKind: 'agent_failure', errorMessage: err.message },
       });
     } catch (publishErr: any) {
       log.error({ error: publishErr.message, runId, agentId }, 'Failed to publish agent_error event');
     }
   }
   ```
   This emits a normal `agent_message` event with a marker text — the transcript will render it in the agent's bubble color so the user sees who failed and continues. We avoid introducing a new event type to keep the SSE/replay code unchanged.

<threat_model>

| Threat | Disposition | Mitigation |
|--------|-------------|-----------|
| Prompt-injection via prior user message text | accept | Same trust model as today — the user's prior messages were already injected into the prompt for the same run; this just consolidates the injection into a labeled block. No new attack surface. |
| Token cost blowup from huge prior conversations | mitigate | Hard caps: 30 events OR 80K chars (whichever hits first). Oldest events drop first. |
| Cross-run leakage via misordered EventStore reads | mitigate | `eq(events.runId, runId)` filter is the one and only scope. New unit test asserts cross-run isolation. |
| Loop crash if eventStore.list fails | mitigate | Wrap the history fetch in try/catch — on failure, log and proceed with empty history (graceful degradation; agents respond context-blind, but they DO respond). |

</threat_model>
