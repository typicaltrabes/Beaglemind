# Phase 17.1 Plan 07: Continue Conversation + History Load Summary

**DEFECT-17-C fixed: EventStore gains a `list` method, runRoundTable injects prior `agent_message` events into each agent's prompt as a `--- PRIOR CONVERSATION ---` block (capped at 30 events / 80K chars, oldest-first drop), and `sendToAgent` failures now publish an `agent_message` marker with `metadata.errorKind='agent_failure'` so the user sees who failed in the transcript instead of silence.**

## Accomplishments

- **EventStore.list shipped.** New read method on the hub's EventStore: `list(tenantId, runId, opts?: { limit?: number })` returns prior events ordered by `sequenceNumber` ASC, default cap at 30, strict `eq(events.runId, runId)` filter for cross-run isolation. The drizzle implementation queries DESC + LIMIT + reverses to ASC client-side — keeps the most-recent-N when runs are long, hands the caller chronological order so prompts read naturally.

- **runRoundTable now context-aware.** Before the per-agent loop, the function fetches prior conversation events via `eventStore.list`, filters to `agent_message` types, excludes the just-persisted user event (filtered by `currentUserSequence`), and builds a `--- PRIOR CONVERSATION ---` block with each turn formatted as `Speaker: text`. The block is injected once at the top of every agent's prompt (between the system context and the new user line). When the run is brand-new (no prior events), the block is empty and the prompt shape is byte-identical to the pre-17.1-07 contract — no regression for new runs.

- **Hard caps applied: 30 events OR 80K chars.** The DB query caps at 30 events via `.limit(HISTORY_EVENT_LIMIT)`. After that, a `while` loop drops the oldest entry until the joined block is under `HISTORY_CHAR_BUDGET` (80,000 chars). Drop-oldest-first matches how humans skim long threads and bounds the prompt size for the day OpenClaw vision pass-through actually doubles the per-message budget.

- **Agent failures now surface in the transcript.** When `sendToAgent` throws (CLI bridge timeout, SSH error, OpenClaw crash), the catch block publishes an `agent_message` event with `content.text = "(<DisplayName> failed to respond — see logs for details)"` and `metadata.errorKind = 'agent_failure'` + `metadata.errorMessage = err.message`. The transcript renders this in the agent's normal bubble color, so the user sees who failed and the round-table proceeds to the next agent. No new event type — keeps SSE replay / chip rendering / scene grouping unchanged.

- **Graceful degradation around eventStore.list.** The history fetch is wrapped in `try/catch`. If `list` throws (DB connection lost, migrated-away schema, malformed row), the catch logs the error and proceeds with `historyBlock = ''`. Agents respond context-blind for that round but they DO respond — better than silent failure, which is the exact UX miss DEFECT-17-C surfaced. T-17-1-07-04 mitigation.

- **TDD honored on Tasks 1-2.** RED commit lands the failing EventStore.list contract tests (5 cases all fail with `store.list is not a function`); GREEN commit lands the implementation and flips them green. Task 3 added 4 more tests for the runRoundTable behavior; all pass after the routes.ts changes. No REFACTOR cycle needed — implementations landed clean.

- **All gates clean.** `apps/agent-hub` vitest 27/27 (6 files, with `DATABASE_URL=postgres://x REDIS_URL=redis://x`). `apps/agent-hub` tsc clean. `apps/web` tsc clean. `apps/web` vitest 227/227 (no regression).

## Decisions Made

1. **`runRoundTable` exported for direct test access.** The new `routes-history.test.ts` cases need to exercise the function's body (history-block construction, agent_failure publication path) without going through `handleRunStart`'s setActiveRun + user-event-persist setup. Switching from `async function` to `export async function` makes the function part of the package's test surface but doesn't change its runtime behavior or expose it to the HTTP layer (only `handleRunStart` is wired into `index.ts`'s router).

2. **`agent_failure` reuses the existing `agent_message` event type with a metadata flag, NOT a new `MessageType`.** A new event type would require updates to the `MessageType` enum in `packages/shared/src/hub-events.ts`, every SSE replay path that switches on type, every transcript renderer, every scene-grouping rule, and Better Auth / RLS / etc. Reusing `agent_message` + `metadata.errorKind` is a single-point change. Future UI work can read the metadata flag to render the marker with a different visual treatment (e.g., a small warning icon, a different bubble border) if needed — no schema migration required.

3. **`currentUserSequence` is threaded down from `handleRunStart` as the filter key for excluding the current turn from PRIOR CONVERSATION.** `MessageRouter.persistAndPublish` returns the persisted envelope (including `sequenceNumber`); `handleRunStart` already destructures `userEvent.sequenceNumber` and returns it to the HTTP caller. Passing the same number into `runRoundTable` and using it in `.filter((e) => e.sequenceNumber !== currentUserSequence)` is the precise, race-free way to exclude the current turn — far better than fragile content-string matching against `userPrompt`.

4. **History fetch wrapped in try/catch with empty-string fallback (T-17-1-07-04).** If `eventStore.list` throws (DB blip, schema missing, drizzle version drift), we log and proceed with `historyBlock = ''`. Agents respond context-blind for that round but they DO respond — the alternative (round-table aborts, user sees silence) is the exact UX miss DEFECT-17-C surfaced. Better-than-silent is the right ceiling here.

5. **Char budget applied AFTER event-count cap, not in a single pass.** The DB query already limits to 30 events. Then we trim entries from the front of the `lines` array until the joined length is under 80K chars. This is two complementary guardrails: 30 events bounds the typical case (most runs won't hit it), 80K chars handles the pathological case (a single 80K-char message). Drop-oldest-first matches how humans skim threads and keeps the most-recent context.

6. **`runRoundTable`'s new params are placed BEFORE the optional `imageAttachments` so the optional stays last.** TypeScript requires optional args to be trailing. The order is `(runId, tenantId, userPrompt, router, eventStore, currentUserSequence, imageAttachments?)` — eventStore and currentUserSequence are required, so they sit before the optional imageAttachments (which 17.1-03 introduced as optional and stays optional).

7. **Existing 17.1-06 routes.test.ts cases stubbed `eventStore.list` to return `[]`, NOT to a fixture array.** The 17.1-06 tests assert prompt content (e.g., `--- USER ATTACHMENTS ---` appears in the agent prompt). If `list` returned non-empty data, the prompt would also have a PRIOR CONVERSATION block, which would still pass the `.toContain()` assertions but would noisy-up the prompt-shape semantics those tests are pinning. Returning `[]` keeps the tests narrowly focused on the 17.1-06 split (user-visible vs agent-visible prompt) without entangling them with 17.1-07 history logic.
