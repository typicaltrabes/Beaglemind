## Accomplishments

- **Hub schema split: `agentPrompt` + `attachmentIds` are now optional on `RunStartBody`.** When provided, the hub persists the user event with the clean `prompt` (+ attachmentIds in content) and sends `agentPrompt` to the OpenClaw round-table. When omitted, behavior is byte-identical to the pre-17.1-06 contract — every existing caller continues to work.
- **Persisted user event shape: `{ text: string; attachmentIds?: string[] }`.** Text is the user-visible string only; the attachment block + extracted text never reach the persisted event. Existing user events without `attachmentIds` keep working — the field is purely additive.
- **WhatsApp-style transcript chips landed.** A user message bubble with attachments renders the user's text, then a chip stack: image attachments are inline `<img>` thumbnails (max-h-60, lazy-loaded, click → full image in new tab), documents (PDF/DOCX/TXT/MD/etc) reuse the existing `ArtifactCard` (View+Download for previewable mimes, Download-only otherwise). - **Reuse-over-rewrite for the chip surface.** `ArtifactCard` already implemented every behavior needed for documents (PDF iframe preview, DOCX mammoth-to-HTML, Download for everything else). `UserMessageAttachments` synthesizes the event-shaped content object the card expects from a thin metadata fetch — zero duplication of preview/download chrome.
- **New thin metadata endpoint: `GET /api/artifacts/[id]`.** Returns `{ id, filename, mimeType, sizeBytes }`, tenant-scoped via `requireTenantContext` + `getTenantDb`. Deliberately omits `extractedText` and `description` so the canonical extracted PDF text stays server-side (agent-prompt-only), preserving the trust boundary the messages-route projection establishes.
- **Test infrastructure unlocked.** Installed `@testing-library/react` + `happy-dom` in apps/web — the 13-02 TODO that explicitly deferred component-render tests is now resolved. Per-file `// @vitest-environment happy-dom` opt-in keeps the default `node` env so pdf-parse tests don't regress.

## Decisions Made

1. **Path B (hub-side `attachmentIds` field) over Path A (web persists event itself).** The plan offered both — Path B is the right one because the hub is the sole writer of events through the SequenceCounter (D-08). Splitting that responsibility (web persists user event + hub persists agent events) would re-open the persist-before-broadcast invariant for re-litigation and create two callers that need to coordinate sequence allocation. The optional widening on `RunStartBody` is additive and backward-compatible by construction.

2. **Per-file vitest environment via annotation, NOT global happy-dom.** The first attempt set `environment: 'happy-dom'` globally; this regressed `lib/extract-attachment.test.ts` (pdf-parse's PDFWorker setup throws under happy-dom — fake-worker code paths reach for Node-only APIs). Reverted to `environment: 'node'` and added `// @vitest-environment happy-dom` to the new component test file. Documented explicitly in vitest.config.ts so future authors know which environment a new test file should opt into.

3. **GET /api/artifacts/[id] excludes `extractedText` and `description`.** Those fields are server-side-only — they exist solely to thread into the `--- USER ATTACHMENTS ---` block in the agent prompt. Exposing them via a client GET would let any authenticated browser dump the canonical PDF text after upload, bypassing whatever prompt-shaping logic the hub adds in the future. Same trust stance as why the SELECT projection in messages/route.ts pulls them but the GET endpoint omits them.

4. **`useArtifactMetadata` uses a module-level Map cache, not react-query / context.** Artifact metadata is immutable post-upload; one fetch per id per session is the right semantics. React-query would add invalidation / refetch / stale-time machinery this surface doesn't need, and it'd drag a provider dependency into the transcript renderer. Module-level Map mirrors the singleton pattern already established by `litellm-client.ts` (lazy module-level client) — single source of truth, test-resettable.

5. **`_resetArtifactMetadataCacheForTest` exported but not used in this plan's tests.** The current 5 cases mock `useArtifactMetadata` directly so the cache is bypassed. The reset helper exists for future tests that exercise the hook with `fetch` mocks (e.g. testing the cache hit path). Mirrors `resetRateLimiterForTest` precedent from earlier phases.

6. **Synthesized fake `HubEventEnvelope` for `ArtifactCard` reuse rather than refactoring `ArtifactCard` to take a metadata-shaped prop.** The card's content shape is `{ filename, mimeType, sizeBytes, artifactId }` — the metadata hook returns `{ filename, mimeType, sizeBytes, id }`. One field rename + the cast is cheaper than refactoring the card to accept both shapes (which would reopen every existing call site). The `as unknown as HubEventEnvelope` cast is documented in a comment and bounded to this single call site.

- **T-17-1-06-04 (Persisted event shape change breaks SSE replay or run-history rendering):** mitigated. Older user events without `attachmentIds` flow through the same `<p>{content.text}</p>` render path as before. No SSE replay code touches the content envelope shape (it's pass-through).
