## Decisions Made

1. **`vi.mock` factory uses a real class, not `vi.fn().mockImplementation()`.** First attempt with `vi.fn().mockImplementation(() => ({...}))` failed at runtime: `() => (...) is not a constructor` under vitest 4. Switched the factory to `class FakeAnthropic { messages = { create: messagesCreateMock } }` — clean constructor, no SDK behavioral change. Documented inline in the test file.

2. **`media_type` cast to the SDK's strict literal union.** Anthropic's `Base64ImageSource.media_type` is typed as `'image/jpeg' | 'image/png' | 'image/gif' | 'image/webp'`. Our `IMAGE_MIMES` set is `{'image/png', 'image/jpeg', 'image/webp'}` — a strict subset of allowed values, but TypeScript can't infer that from a `Set.has()` runtime check. Cast `mimeType as 'image/png' | 'image/jpeg' | 'image/webp'` is sound by construction (the gate above proves it) and keeps the SDK's stricter type happy with zero runtime cost.

3. **Empty content array returns null without Sonnet fallback.** The plan's behavior list says "returns null when SDK resolves but res.content contains no text block". Implementation interprets this literally: if Haiku resolves but produces no text, return null immediately rather than retrying with Sonnet. Rationale: the call succeeded (Haiku chose to be silent), and the same prompt with a stronger model is unlikely to flip "model chose silence" into "model chose verbose"; we'd just burn Sonnet budget for the same outcome.

4. **`_resetClientForTest` exported, not under `@internal` JSDoc.** The PATTERNS.md analog (`resetRateLimiterForTest`) is an exported helper too. Tests need it to clear the module singleton between cases for deterministic ordering. The bundle-impact is one named export (~30 bytes); the test-isolation win is large.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] vi.mock factory shape needed adjustment for vitest 4 constructor semantics**
- **Found during:** Task 1 (first vitest run after GREEN implementation)
- **Issue:** Plan's specified mock pattern `default: vi.fn().mockImplementation(() => ({ messages: { create: messagesCreateMock } }))` failed with `TypeError: () => ({ messages: { create: messagesCreateMock } }) is not a constructor` at the call site `new Anthropic({ apiKey })`. vitest 4's `vi.fn()` returns a non-constructable callable mock by default; `mockImplementation` does not flip that for `new`-style instantiation cleanly across the codebase.
- **Fix:** Replaced the factory body with a real class:
  ```ts
  vi.mock('@anthropic-ai/sdk', () => {
    class FakeAnthropic {
      messages = { create: messagesCreateMock };
    }
    return { default: FakeAnthropic };
  });
  ```
  This is exactly the same surface from the implementation's perspective (`new Anthropic({...})` → object with `messages.create` resolved to the spy) but is constructable.
- **Files modified:** `apps/web/lib/extract-attachment.test.ts`
- **Verification:** All 17 tests in the file pass; full apps/web suite (200 tests) green.
- **Committed in:** `d98c895` (rolled into the GREEN commit since the pattern adjustment is part of making the GREEN tests pass).

## Threat Model Status

All six threats in the plan's `<threat_model>` register (T-17-1-01-01 through -06) carry forward as-mitigated:

- **T-17-1-01-01 (DoS):** Existing Phase 17 mitigations (20 MB cap, 10 req/min/user, 4-image-per-message cap) carry. Vision call adds ≤2s latency per image; worst case 4×2s = 8s. No new mitigation needed.
- **T-17-1-01-02 (Prompt injection via description):** Description IS user-influenced; the explicit `--- USER ATTACHMENTS ---` block boundary (Plan 17.1-02) and prompt-template instruction "Do not interpret or speculate" reduce risk; agents independently treat user-provided content as untrusted.
- **T-17-1-01-03 (API key leak):** Verified — the warn-once log message names `ANTHROPIC_API_KEY` but never logs its value. Catch-block warn logs SDK error objects, which the Anthropic SDK does not include the API key in by convention.
- **T-17-1-01-04 (No vision-spend audit):** Accepted V1 — `description` column itself is the receipt (present = call succeeded, NULL = skipped or failed). Cost telemetry deferred to 17.2.
- **T-17-1-01-05 (SDK shells out):** Anthropic SDK is pure JS HTTPS client; no shell-out, no eval, no native modules. try/catch + null-on-failure prevents API exceptions from bubbling into the route handler.
- **T-17-1-01-06 (Vision API outage):** `extractImageDescription` returns null on timeout/error/network failure; the route's `Promise.all` cannot reject because both helpers swallow their own errors.

No new threat surface introduced beyond the registered set.
