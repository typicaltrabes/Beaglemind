## Decisions Made

1. **Single helper file shared between composer and route.** Two separate `EXT_TO_MIME` maps would re-create exactly the kind of allowlist drift that produced DEFECT-17-A. The helper is in `apps/web/lib/` (not `packages/shared/`) because the consumers are both in `apps/web` and there's no third consumer on the horizon — promoting to `shared` is a YAGNI move.

2. **`ALLOWED_MIMES` derived from `EXT_TO_MIME` values, not hand-maintained.** Adding a new supported extension automatically widens the allowlist Set; the two structures cannot drift. The same mime can map from multiple extensions (`jpg` and `jpeg` both → `image/jpeg`) and `Set` deduplicates naturally.

3. **`UNRELIABLE_BROWSER_MIMES` whitelist of two specific values, not a blanket "always fall back to extension".** Browser-supplied `file.type` is reliable for the vast majority of uploads — Mac PNG uploads, Linux PDF uploads, etc. Forcing extension fallback on every upload would mean a maliciously renamed `evil.exe → evil.png` would pass the gate based on the extension alone (already accepted in Phase 17's threat model, but explicitly making it the default path raises the surface). Narrowly enumerating `''` and `application/octet-stream` keeps the Phase 17 trust model intact while patching the specific Windows-MD case.

4. **Persist `resolvedMime` to `artifacts.mime_type`, not raw `file.type`.** This was the subtle half of the fix: even if the validation gate passes via extension fallback, persisting the empty `file.type` would mean `extractAttachment`'s mime switch (`if (mime === 'text/markdown') ...`) misses, the `attachment-block.ts` image-vs-PDF branching mis-routes, and the chip's filename → icon mapping in the transcript renders the generic placeholder. Persisting the canonical value once at the boundary keeps every downstream consumer honest.

5. **Route-level test uses a real RFC 4122 v4 UUID, not a placeholder.** Zod v4's `.uuid()` format check is strict on version + variant nibbles. Initial test with `'00000000-0000-0000-0000-000000000001'` failed the format and the route returned a 400 with the Zod error message — masking the actual mime-validation behavior I was trying to test. Switched to `'11111111-2222-4333-8444-555555555555'` (valid v4 layout). Documented as a code comment so future test authors don't re-trip the same wire.

**1. [Rule 3 — Blocking] Route-level vitest test UUID failed Zod's strict v4 format check**
- **Found during:** Task 4 (first run of the new route.test.ts)

- **Fix:** Replaced with a real RFC 4122 v4 UUID `'11111111-2222-4333-8444-555555555555'` (version=4 in the third group, variant=8 in the fourth group). Added a code comment to flag the constraint for future authors.

## Threat Model Status

The plan's `<threat_model>` register lists three rows; all carry forward as-mitigated:

- **Extension-spoofing (`evil.exe` renamed to `evil.md`):** Disposition = `accept`. The bytes-level extraction path for `text/markdown` is `utf-8 read` (byte-safe; worst case is garbage characters in the agent prompt, no code execution). Same trust model as Phase 17's Track 1 attachment intake.
- **Allowlist bypass via case manipulation:** Mitigated by `mimeFromExtension`'s `.toLowerCase()` on the extension before lookup; `EXT_TO_MIME` keys are all lowercase. Verified by the `'NOTES.MD' → 'text/markdown'` and `'photo.PNG' → 'image/png'` test cases.
- **Empty filename:** Mitigated by `mimeFromExtension('')` returning null (the `lastIndexOf('.') < 0` branch); the route's `if (!resolvedMime)` gate then returns 400. Verified by the `'noextension'` test case.

No new attack surface introduced beyond Phase 17's existing register. The fix only widens detection of the same allowed mimes; it does NOT add new mime types.
