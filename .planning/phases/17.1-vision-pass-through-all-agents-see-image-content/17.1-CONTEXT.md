# Phase 17.1: Vision Pass-Through — All Agents See Image Content — Context

**Gathered:** 2026-04-28
**Status:** Ready for planning
**Source:** Lucas's UAT call after Phase 17 V1: "we need to build this product to be able to have all the agents see images and attachments even if they are on weaker LLMs as that will eventually likely be our route."

<domain>
## Phase Boundary

Phase 17 V1 prepended a placeholder `(image — included with this message)` to the agent prompt for image attachments. UAT proved this is insufficient: agents acknowledge the file exists but can't reason about content. Lucas's product call: every agent must engage with image content, regardless of whether the model is vision-capable.

This phase adds two complementary mechanisms:

**Track 1 — Server-side image description (description floor for ALL agents).**
At upload time, when the mime is an image (`image/png`, `image/jpeg`, `image/webp`), the `POST /api/runs/[id]/attachments` route calls the Anthropic vision API once and persists a written description of the image. This description replaces the V1 placeholder in the prepended `--- USER ATTACHMENTS ---` block, so every agent (Mo, Jarvis, Herman, Sam, future weaker models) sees a meaningful textual rendering of every image.

**Track 2 — Image bytes through the CLI bridge for vision-capable agents (vision ceiling for top agents).**
Vision-capable agents (Mo + Jarvis on Anthropic Opus) ALSO receive the image bytes through OpenClaw's CLI bridge so they can pick up details the description missed. Non-vision agents (Herman now, future weaker models) get only the description — no degradation, no special-casing in the prompt.

Together: every agent has a floor (description); top agents have a ceiling (actual pixels).

**What this phase does NOT do:**
- OCR / handwriting transcription of images. The vision-API description is what we ship; agents don't get raw OCR.
- Per-agent description customization (every agent sees the same description).
- Audio / video uploads (still out of scope, deferred).
- Asynchronous description generation — V1 caps at 4 images per message and runs synchronously in the upload route. Acceptable latency: ~1-3 sec/image.
- Re-describing images after Phase 17 already uploaded them — only NEW uploads from 17.1 onward get descriptions. Phase 17's existing artifacts keep `description = NULL`.
- Routing through LiteLLM. LiteLLM is currently down (Henrik's fix pending) — 17.1 calls Anthropic SDK directly. Phase 18+ can decide whether to migrate to LiteLLM later.

</domain>

<decisions>
## Implementation Decisions

### Track 1 — Vision description at upload time

- **New column:** `artifacts.description TEXT NULL`. Idempotent migration `migrate-17-1.ts` (template: `migrate-17.ts` from Phase 17). `ADD COLUMN IF NOT EXISTS description TEXT`.
- **Drizzle schema update:** `description: text('description')` added to the artifacts schema in `packages/db/src/schema/tenant.ts`.
- **Vision API:** Anthropic SDK direct (NOT LiteLLM — it's down). Use `@anthropic-ai/sdk` (already in repo if not, install). Model: `claude-haiku-4-5-20251001` (cost: ~$0.003 per image; speed: ~1-2s). Falls back to `claude-sonnet-4-6` if Haiku errors twice in a row.
- **Prompt for vision call:** Single system+user message: `"Describe this image factually in 2-4 sentences. Note key visual elements, text content if any, and overall composition. Do not interpret or speculate — just describe what's visible."` Max output: 300 tokens.
- **Where it runs:** `apps/web/lib/extract-attachment.ts` gains an `extractImageDescription(buffer, mimeType): Promise<string | null>` function. Called from the existing extraction pathway when mime is image/*. PDF/DOCX/TXT/MD paths unchanged.
- **Synchronous in upload route:** acceptable for V1 (4-image cap → max ~12s extra latency in worst case; users see the chip stuck on "uploading…" until done, which is correct UX feedback).
- **Failure mode:** if the vision call fails (API timeout, invalid API key, Anthropic outage), log a warning, set `description = NULL`, return success on the upload. The image is still stored in MinIO and the artifact row is created — agents will receive a textual fallback `(image attached — description unavailable)` so they at least know an image was attached.
- **Caching:** none in V1. Re-uploading the same image generates a fresh description. Add caching (by sha256) in a future phase if cost becomes a concern.
- **Env var:** `ANTHROPIC_API_KEY` — must be set in `/opt/beagle-console/.env`. If missing, vision call short-circuits to NULL with a startup warning.

### Track 2 — CLI bridge image pass-through

- **Hub schema extension:** the `RunStartBody` / per-agent message payload gains an optional `imageAttachments: Array<{ filename: string; mimeType: string; base64: string }>` field. Vision-capable agents receive this; non-vision agents do not.
- **Vision-capability flag:** new field `visionCapable: boolean` on `agentConfig` in `apps/web/lib/agent-config.ts`. Initial values: Mo=true, Jarvis=true, Herman=false, Sam=false. Hub uses this when fanning out the round-table to decide who gets bytes.
- **CLI bridge change:** `apps/agent-hub/src/connections/openclaw-cli-bridge.ts` accepts `imageAttachments` and writes each image's base64 to a temp file (`/tmp/console-vision-${uuid}.${ext}`), then invokes OpenClaw with the appropriate vision flags (TBD by researcher — likely `--image /tmp/...` or `--attachment`). Temp files cleaned up after the agent call returns.
- **Web → hub plumbing:** when the messages route forwards a request to the hub, if any attachment in `attachmentIds` has `mimeType` starting with `image/`, fetch the bytes from MinIO, base64-encode, include in the hub payload alongside `attachmentIds`. Don't re-fetch bytes per agent — fetch once in web, pass to hub, hub fans out.
- **Image size guard:** if total image bytes exceed 10 MB across all attachments in a message, log a warning and skip image bytes (description still goes through). Prevents blowing up the OpenClaw command line on huge uploads.

### Track 3 — Prompt block update

- **`apps/web/lib/attachment-block.ts` changes:** when building the block for an image attachment, replace the current `(image — included with this message)` placeholder with the actual `description` value from the artifact row. Format:
  ```
  [2] screenshot.png (PNG, 38 KB)
  Description: A screenshot of a dashboard showing four KPI cards with numeric values and a sidebar with agent names. The amber accent color is visible in the header wordmark.
  ```
- **Fallback:** if `description` is NULL (vision call failed or row predates 17.1), use `(image attached — description unavailable)`.
- **Test coverage:** extend `apps/web/lib/attachment-block.test.ts` with cases for: image with description, image with NULL description, mixed PDF + image with description.

### Cross-cutting
- **Single deploy at end of phase**, same workflow as Phase 17.
- **Atomic commits per track** — `feat(17.1-NN):` scope.
- **[BLOCKING] schema-push ordering:** migrate-17-1.ts runs BEFORE container rebuild. Same pattern as Phase 17.
- **No new top-level UI surface.** Composer behavior unchanged; only the agent-side experience changes.
- **Pre-flight:** typecheck + vitest before deploy.
- **LiteLLM:** still down on Henrik's end. 17.1 uses Anthropic SDK direct. Document this as a known dependency-minimization choice — Phase 18+ can route through LiteLLM if it's revived.

### Claude's Discretion
- Exact OpenClaw flag for image attachments — researcher must verify against OpenClaw CLI docs / source. If OpenClaw doesn't accept image flags directly, the executor may need to send the image as part of the message body (base64-embedded) or use a sidecar mechanism. Researcher should confirm before planning.
- Whether to add a small async-vs-sync toggle for the vision call (e.g., feature flag). Default: synchronous, no flag, V1 simplicity.
- Specific prompt tuning for the vision-API description — start with the prompt above, iterate if outputs are too verbose / not factual enough during UAT.

- **UAT-17-1-06**: User sends a follow-up message (with or without attachments) to a run whose status is `completed`; agents respond in the same round-table format AND demonstrate awareness of prior conversation context (reference earlier messages by content). Run status flips back to `executing` for the duration of the new round, then to `completed` again.

<addendum date="2026-04-29">
## Phase 17 UAT Defect Rollup

Lucas's Phase 17 UAT (2026-04-29) flagged three defects. Per Lucas's call, all three are folded into Phase 17.1 — single deploy, no separate hotfix phase. Phase 17 stays open until 17.1 ships and re-tests UAT-17-01..03 alongside the new UAT-17-1-04..06.

### DEFECT-17-A — `.md` files rejected on Windows
**Symptom:** Lucas couldn't upload `.md` files. Spec says MD is allowed.

**Root cause:** Both the composer (`apps/web/components/transcript/composer.tsx:183`) and the upload route (`apps/web/app/api/runs/[id]/attachments/route.ts:71`) gate on `file.type` against an `ALLOWED_MIME` set. On Windows (and some browsers), `.md` files come through with `file.type === ''` because `text/markdown` isn't registered in the OS MIME database. The check fails and the file is rejected as `Unsupported file type`.

**Fix decision:** Add a file-extension fallback. If `file.type` is empty OR not in `ALLOWED_MIME`, derive mime from the filename extension (`.md` → `text/markdown`, `.txt` → `text/plain`, `.pdf` → `application/pdf`, etc.). Apply the same mapping in both client (composer) and server (route). When the server accepts based on extension, persist the derived mime to `artifacts.mime_type` so the rest of the pipeline behaves consistently.

**Plan:** 17.1-05.

### DEFECT-17-B — Extracted text dumps into transcript bubble
**Symptom:** When a user attaches a PDF, the entire extracted text content (potentially thousands of chars) renders inside the user's message bubble in the transcript. With chart-heavy PDFs the result is "very ugly". Lucas's product call: WhatsApp-style — show a clickable chip in the bubble, not the text.

**Root cause:** `apps/web/app/api/runs/[id]/messages/route.ts` builds `attachmentBlock + content` and forwards it to `hubClient.startRun({ prompt: ... })`. The hub then persists a single `agent_message` event with `agentId='user'` and `content.text` containing the WHOLE prepended string (attachment block + user text). The transcript's `AgentMessage` component renders `content.text` verbatim → text dump.

**Fix decision:** Separate "agent prompt" from "user-visible content".

- **Hub schema (`RunStartBody`):** add an optional `agentPrompt: string` field. When provided, the hub uses `agentPrompt` as the OpenClaw input but persists the user event with `prompt` (the user-visible text) as `content.text`. When omitted, behavior is unchanged (backward-compatible).
- **Web messages route:** persist behavior — pass `prompt: content` (user text only) and `agentPrompt: attachmentBlock + content` (with the attachment block). Also include `attachmentIds: string[]` in the persisted user event's `content` so the transcript can render chips.
- **User event content shape:** `content: { text: string; attachmentIds?: string[] }`. Existing user events without attachmentIds keep working unchanged.
- **Transcript rendering:** `AgentMessage` (or a new `UserMessage` subcomponent) detects `agentId === 'user'` AND `content.attachmentIds?.length > 0` and renders the existing `ArtifactCard` component for each attachment below the text. `ArtifactCard` already handles PDF preview, DOCX preview via mammoth, and download — reuse it. Add an inline-thumbnail variant for images (`mimeType.startsWith('image/')`) that renders an `<img>` from `/api/artifacts/<id>/download` with `loading="lazy"` and a max-height cap (e.g. 240px); clicking opens the full image in a new tab.

**Out of scope for this plan:** rewriting `ArtifactCard` itself, sharing/permissions changes. Reuse only.

**Plan:** 17.1-06.

### DEFECT-17-C — Agents silent on completed runs
**Symptom:** Lucas added attachments to an already-completed run, sent the message, no agents responded. Lucas's product call: a chat is a chat — agents must always answer follow-ups inside a run's context regardless of lifecycle status.

**Root cause:** Two layered problems —
1. The web messages route at `apps/web/app/api/runs/[id]/messages/route.ts:108-112` already flips `status` from `completed` back to `executing` and calls `hubClient.startRun()`. So that path looks correct on web. The silence likely stems from `runRoundTable` in `apps/agent-hub/src/http/routes.ts:155-212` running fine but with ZERO awareness of prior conversation — `transcript: string[]` is local to the function and starts empty every call. With huge attachment-prepended prompts this can also cause CLI bridge timeouts that fail silently per the catch-all at line 209-211.
2. Even on success, agents respond context-blind — they'd answer "What's in this PDF?" with no idea what the run was previously about. That's the bigger UX miss.

**Fix decision:**

- **Load conversation history into the hub round-table prompt.** When `runRoundTable` fires for a runId that already has prior `agent_message` events in `events`, fetch those events (ordered by `sequenceNumber`) BEFORE the new user prompt and inject them as `--- PRIOR CONVERSATION ---` ... `--- END PRIOR CONVERSATION ---` between the system context block and the new user prompt. Cap at the most recent N events (default 30) or M total characters (default 80K) — whichever hits first — to bound prompt size.
- **EventStore gains a read method.** Add `EventStore.list(tenantId, runId, opts?: { limit?: number })` that returns events for the run ordered by `sequenceNumber`. Used by `runRoundTable` to load history before each round. (Existing tenant `events` table already has the data — no new schema.)
- **Failure surfacing.** When `sendToAgent` throws inside `runRoundTable`, log at `error` level AND publish an `agent_error` event to the EventStore so the user sees something in the transcript instead of silence. (Lightweight: just a one-line note like "Mo failed to respond — see logs". Not fancy.)
- **No new web-side gating.** The composer already permits send on completed runs (`composer.tsx:103-105`); the messages route already flips status back to executing. Both stay as-is. Fix is hub-side.

**Out of scope:** archived/deleted runs (treat as still completed), per-agent context caps, vector retrieval for very long histories. Future phases.

**Plan:** 17.1-07.
